## 0. アプリの一行説明

**「指定 5 場の今日の荒れレースだけを見つけて、 5,000 円でどう買うかを提示するアプリ」**

ユーザーは何も入力しない。 アプリ側がすべて決める。

### 0.1 思想 (両立すべき 2 つ)
- **初心者は超簡単に使える** (1 秒で勝負あり/なし、 3 秒で買い目)
- **上級者は深掘りできる** (折りたたみ + 研究所タブ)

「シンプル＝情報不足」 にしてはいけない。 ホーム画面で **買い・見送り・AI 評価・理由** まで即見える状態にする。 「なぜ買いなのか」 を 2 タップ以上奥に隠すのは**禁止**。

## 4. トップ画面 (ホーム) に出すもの

### 上部 (ヘッダー — 最小限)
- 🌊 万舟研究所 (ロゴ・黄色)
- 「🔄 今すぐ更新」 ボタン
- 今日の累計収支 (例: `+12,000円` / `−5,000円`)
- ログイン状態アイコン (Supabase 同期確認用 — 小さく)

### 中央 (今日の勝負)

#### 勝負レースが 1 件以上ある場合
レースカードを 1 件ずつ縦に並べる。 各カードには **以下 7 項目すべてが折りたたまずに即見える状態** で並ぶ:
1. **対象場** (戸田・江戸川…)
2. **レース番号** (◯R)
3. **締切までの時間** (例: 「あと 7 分」 ・5 分以内は赤点滅)
4. **判定** (🚨 激荒れ警報 / ⚠️ 荒れ注意 / 🔵 見送り推奨)
5. **荒れスコア / AI 評価** (◯◯/100 + 「強気」「弱気」 等の言葉)
6. **5,000 円の買い目** (例: 「3連単 1-2-4 / 1-2-3 各 2,500 円」)
7. **理由 3 行** (なぜ荒れるか・誰が来そうか・なぜこの買い目か)

⚠️ 重要: 6 と 7 を「詳しく見る」 ボタンの中に隠すのは**禁止**。 必ず最初から見える。

#### 勝負レースが 0 件の場合
- 大きく「**今日は勝負なし**」 (画面中央)
- サブテキスト: 「5 場で今日 N レース監視しましたが、 荒れ条件が揃ったレースはありませんでした」
- 何もしないことを肯定する文言 (「無理に買わない方が研究所として正しいです」)

### 下部 (各カードの中・折りたたみ)
タップで開閉する詳細セクション:
- スコア内訳 (進入不安/強風波/1号艇不安/攻め手/展示異変/オッズ妙味 6 軸)
- 気象・水面 (風速・波高・天候)
- 公式リンク (出走表 / オッズ / 直前情報 / 結果)
- レース終了後: 結果 (1 着 2 着 3 着 + 配当)
- レース終了後: 5,000 円買って当たったか・収支

## 5. やらないこと (絶対禁止)

| やらないこと | 理由 |
|---|---|
| 金額をユーザーに入力させる | 設定アプリにしないため |
| 予想スタイル 3 択 | 既に削除済み (Round 168) |
| 全 12 レース機械的表示 | 5 場・75 点+ のみ |
| 対象外場 (大村・住之江 等) の表示 | 5 場限定の徹底 |
| 「📡 監視中: N レース」 数字バッジ | 「全レース見てる」 誤解の元 |
| 「🤐 見送り N 件」 数字バッジ | ユーザー視点で意味不明 |
| 「😱 万舟見逃し N 件」 数字バッジ | 心理的ノイズ・ユーザー向けでない |
| 「💰 リアル/🧪 エア」 切替ボタン | 5,000 円固定なので不要 |
| 「💰 N 件 buy」 件数バッジ | 件数より「今日勝負あり/なし」 が大事 |
| 設定画面を最初に見せる | 開いた瞬間に勝負が分かるべき |
| 自動ログイン・自動購入 | 全プロジェクト禁止事項 |

## 6. タブ構成 (画面下固定 4 タブ)

| タブ | 名前 | 内容 | 入口の大きさ |
|---|---|---|---|
| 🏠 | ホーム | 上記 4 章 (厳選された勝負レースカード or「今日は勝負なし」)。 **これがメイン** | 大 |
| 📋 | 一覧 | 5 場全レースの予想 (荒れスコア + 買い目 + 締切) を 1 行ずつ横並びで確認 (Round 190.5 復活) | 中 |
| 🔬 | 研究所 | 上級者向け深掘り画面。 詳細は §6.1 参照。 **入口は小さく** (一般ユーザーを迷わせない) | 小 |
| ⚙️ | 設定 | 4 項目だけ (詳細は §6.2)。 大量の設定項目を置くのは禁止 | 小 |

**Round 190.5 メモ**: 旧「📋 一覧」 (Round 171 で削除) は「全 12 レース機械的並べはダメ」 という理由で廃止していたが、 「予想していないように見える」 という致命的なユーザー体験を生んでいた。
新版は **5 場限定・荒れスコアつき・買い目つき** で「全件をパッと確認できる軽量ビュー」 として復活。 「厳選はホームの責務、 一覧は確認の責務」 と棲み分ける。

### 6.1 「🔬 研究所」 タブの中身

99% のユーザーは研究したくない。 研究所は**深掘りしたい人専用**。 「研究所をメインにしない」 こと。

#### 6.1.1 中身一覧
- **担当 5 場ごとのリアルタイム分析ランキング** (詳細は §6.1.2 / Round 178+ で実装)
- **AI 根拠** (なぜこの判定か・スコアの内訳・どの要因が効いたか)
- **学習データ** (見送り正答率・取りこぼし額・荒れスコア重み調整)
- **モーター傾向** (場別モーターの今節成績 — ランキングと統合可)
- **会場傾向** (5 場それぞれの今節の決まり手分布・配当傾向)
- **万舟分析** (過去の万舟出現パターン・期待値分析)

#### 6.1.2 担当 5 場ごとのランキング機能 (信頼感の中核)

**目的**: 「このアプリ、 本当に研究してる」 という信頼感を生む。 見せかけ禁止。

##### 設計原則 (絶対守る)
- 5 場リストは **動的取得** (`mansyu.js:TARGET_VENUES` を参照)。 場名をコードに固定埋め込み**禁止**
- **全場混合ランキング禁止** — 5 場それぞれで独立に TOP10 を計算
- 単純な勝率順ではなく **「最近伸びている」「人気の割に強い」「今節気配が良い」 を加味**
- 各順位に **「なぜ上位か」 の一言タグ**を必ず付ける (一瞬で読める)
- **AI 評価ロジックと連動** (見せかけのスコアを別計算するのは禁止 = 予想と整合する数字を出す)

##### モーター TOP10 (場別)
評価軸:
- 2 連率
- 3 連率
- 直近成績 (今節 / 前節)
- 展示気配 (展示タイム偏差・部品交換歴)
- 上昇傾向 (前節→今節のスコア差)
- AI 評価 (上記を重み付き合算)

各艇に付ける一言タグの例:
- 「展示気配◎」 (展示タイム上位)
- 「今節伸び型」 (節を追うごとに成績向上)
- 「部品交換ハマり」 (直近の交換が功を奏している)
- 「人気の割に強い」 (人気度に対して連対率が高い)

##### 選手 TOP10 (場別)
評価軸:
- ST (今節平均・コース別)
- コース別成績 (1 コース～6 コース別の連対率)
- モーター相性 (現在乗っているモーター × 選手の組み合わせ実績)
- 近況 (直近 30 日の連対率・トレンド)
- **会場相性** (その場での過去成績 — 全国成績ではなくローカル成績重視)
- 直近数節 (前節・前々節の成績推移)
- AI 期待値 (上記を重み付き合算)

各選手に付ける一言タグの例:
- 「ST 安定」 (今節 ST が上位)
- 「相性良好」 (このモーターでの過去成績が良い)
- 「イン巧者」 (1 コースでの連対率が高い)
- 「マクリ得意」 (アウトコースから差し込む型)
- 「会場巧者」 (全国成績は普通だがこの場では強い)

##### UI 原則
- **ごちゃごちゃ禁止** (情報密度を上げすぎない)
- **色を使いすぎない** (黒・濃紺ベース + アクセント 1-2 色のみ)
- **一瞬で読める** (1 行 1 艇 / 1 選手・場名タブで切替・タグは 1 個)
- **ホーム画面を重くしない** (このランキングは 🔬 研究所タブ内に集約。 ホームには出さない)
- 入口は小さく (一般ユーザーを迷わせない)

### 6.2 「⚙️ 設定」 タブの中身 (4 項目だけ)

| 項目 | 必須度 | 理由 |
|---|---|---|
| ログイン (Supabase) | 必須 | 端末間同期 |
| ログアウト | 必須 | アカウント切替・離脱 |
| データ削除 (リセット) | 必須 | 緊急時の全消去 |
| 通知 ON/OFF | 必須 | 「今ここだけ勝負」「期待値急上昇」「危険レース」 通知の土台 |

⚠️ 禁止: 資金・1 日予算・1 レース上限・スタイル選択・セーフティ買い ON/OFF など、 思想を崩す設定は**追加しない**。

## 8. 100 点の定義 (受け入れ基準)

このすべてが満たされて初めて「100 点」と認める:

- [ ] アプリを開いて **1 秒以内**に「今日勝負あり / なし」 が分かる
- [ ] 勝負ありの場合、 各レースの **対象場・R・締切・判定・スコア・買い目・理由 3 行** が **3 秒以内**に視認できる (折りたたみの中に隠さない)
- [ ] 理由は **3 行以内**で、 専門用語抜きで読める
- [ ] ユーザーは **何も入力しなくていい** (利用同意の初回チェック以外)
- [ ] **設定画面を 1 度も開かなくていい** (通知 ON/OFF だけ任意で開く)
- [ ] **金額・スタイル・場の選択肢が画面のどこにも無い**
- [ ] スマホで **片手・1 タップ**で各レースの詳細が開閉する
- [ ] PC でも **2-3 列のダッシュボード**として情報密度が上がる
- [ ] データ取得失敗時に **「⚠️ 公式サイトから取れませんでした」** の状態が明示される
- [ ] 黒・濃紺ベース、 赤・黄アクセントで「研究所感」 がある (色数 3 系統以下)
- [ ] **対象外場 (大村・住之江 等) のレースが画面のどこにも出ない**
- [ ] **シンプルだが情報不足ではない** — 「なぜ買いか」 「なぜ見送りか」 がホーム画面で読める
- [ ] **研究所タブの入口が小さい** (一般ユーザーが迷わない位置・サイズ)
- [ ] 上級者は研究所タブで **AI 根拠・モーター傾向・会場傾向・万舟分析** まで深掘りできる

## 12. AI 進化のロードマップ (使うほど賢くなる構造)

shoug 思想:「固定ルールだけのアプリにしない。 使うほど賢くなる AI を作る」 を実現するための段階計画。 一足飛びは不可能 → 3 段階で積み上げる。

### 段階 A — 既存ループを閉じる (短期 / Round 172-172.5)

現状: 統計分析 / 重み補正提案 まで自動。 適用は手動ボタン頼み。 ループが閉じていない。

ゴール: **「分析 → 提案 → 自動適用 → 効果検証 → 自動ロールバック」 の閉ループ**を作る。

実装内容:
- Round 172 前半 — 自動重み適用:
  - mansyuLearning.analyzeMansyuLearning を 1 日 1 回実行
  - 安定性チェック (確定済 ≥ 10 件 / 各成分集計 ≥ 8 件) を通った recommendations のみ採用
  - applyAllRecommendations で重みを更新
  - 適用前後の重み・タイムスタンプ・サンプル数を `mansyuLearningHistory` (localStorage) に記録
  - ホーム上部に小さなバナー「🤖 学習が動きました (◯ 件のデータから重み微調整)」 を表示
- Round 172.5 後半 — 自動ロールバック:
  - 適用後 7-14 日の「見送り正答率 / 見立て正答率」 を測定
  - 悪化 (例: −5% 以上) していたら前の重みに自動復元
  - 復元理由も `mansyuLearningHistory` に記録
  - 安全装置: 同じ提案を 3 回連続で却下されたら「学習停止」 状態に

成果物:
- localStorage キー: `mansyuLearningHistory` (適用 / 却下 / 復元の履歴 直近 30 件)
- 既存の `mansyuWeights` を機械的に書き換え
- 既存の `learning.js` (旧 EV 用) のロジックを参考に、 万舟向け版を新規作成

### 段階 B — 細粒度学習 (中期 / Round 182)

現状: 全データを混ぜて 1 セットの重み。 場 / モーター / 選手 の特性を反映できない。

ゴール: **場別 / モーター別 / 選手別**に重みや評価を持つ。

実装内容:
- 5 場それぞれに別の重みセットを持つ (戸田は風重視、 江戸川は潮重視 等)
- モーター ID ごとの「期待値補正」 を学習 (このモーターは公称 2 連率より実際は強い 等)
- 選手 ID ごとの「会場相性」 を学習 (この選手は江戸川で異常に強い 等)
- ランキング機能 (§6.1.2) と統合 → ランキング上位 = 学習で「強い」 と判定された人
- ストレージ:
  - `mansyuVenueWeights`: { jcd → weights }
  - `mansyuMotorAdj`: { jcd-motorId → adjustment }
  - `mansyuRacerVenueFit`: { racerId-jcd → fit_score }

### 段階 C — ディープラーニング (長期 / Round 200 想定)

現状: 統計集計 + 線形重み合算。 非線形パターン (例: 風速 × ST × 1 号艇クラス の組み合わせで急に荒れる) は捉えられない。

ゴール: **本物の機械学習モデル**で予想する。

選択肢:
- 案 1: TensorFlow.js (ブラウザ内推論) — レスポンス速いが学習はサーバ必須
- 案 2: Vercel Functions + Hugging Face / OpenAI API 経由でサーバ推論
- 案 3: Python (Vercel Python Runtime or 別ホスト) で scikit-learn / XGBoost

実装前提:
- 段階 A-B で学習データ (`mansyuSkipLog`) が 1,000 件以上溜まっている
- 「入力特徴量 → 結果 (荒れたか / 配当)」 の教師データセットが構築できている
- shoug さんが本気でディープラーニングをやる意思を示している (運用コスト / 維持工数を許容)

⚠️ 段階 C は非常に大規模。 段階 A-B が安定運用できてから着手。 場合によっては SPEC を改訂して別アプリ化も検討。

### 段階 A-C 共通の原則

- **過学習防止**: 7/14/30 日の 3 期間で安定した傾向だけ採用 (既存 learning.js の方針を継承)
- **少数サンプル禁止**: 10 件未満の傾向では学習しない
- **ロールバック必須**: 学習後に悪化したら必ず前の状態に戻る
- **学習ログを残す**: 「いつ」「何を」「なぜ」 適用 / 却下 / 復元したかを記録 → ユーザーが研究所タブで見れる

## 13. 自己学習 AI の安全設計 3 層 (shoug 指示・必須要件)

「使うほど賢くなる AI」 を本気で作るなら、 以下の 3 層が**必須**。 これがない自己学習 AI は危険 (= 学習が暴走して逆に悪化する可能性)。 SPEC §12 段階 A/B の上に追加する安全装置。

### 13.1 第 1 層: 学習データの完全保存 (Round 185)

**ゴール**: 予想時点のすべての情報を残し、 後から「なぜこの予想をしたのか」 を完全に再現できるようにする。

#### 保存対象 (mansyuSkipLog の各エントリに `snapshot` フィールド追加)
- ✅ 既存: レース基本情報 / AI スコア + 6 成分 / 判断 (show/skip) / 結果
- ➕ 追加:
  - `boats[]` 6 艇分のフルスナップショット (racer / class / winRate / localWinRate / motor2 / boat2 / ST / 展示タイム / 部品交換)
  - `apiOdds` オッズ詳細 (3連単 / 単勝)
  - `weather / wind / wave / windDir` 気象・水面
  - `reasons[]` 判断理由テキスト配列
  - `buyOrders[]` 5,000 円配分済み買い目
  - `virtualPnl` 仮想収支 (5,000 円買って当たれば配当 - 5,000 円、 外れれば -5,000 円)

#### ストレージ設計
- localStorage キー: `manfuneSkipLog` (既存) を拡張
- 30 日保持 (pruneOld 既存)
- 容量目安: 1 エントリ 5-10 KB → 30 日 × 200 件 = 1-2 MB (5 MB 制限内)
- 容量超過時: skip 判定エントリの snapshot を削除して show 判定だけ残す

### 13.2 第 2 層: バックテスト機能 (Round 186)

**ゴール**: 新しい荒れスコア計算式や重み調整を、 過去データで先に検証する。 「感覚で良くなった」 は禁止。

#### 必要な機能
- 過去 mansyuSkipLog の snapshot.boats / apiOdds / weather から `scoreMansyu()` を**再計算**
- 「現在の重み」 vs 「テスト中の重み」 で全エントリを再評価
- 集計指標:
  - **的中率**: show 判定したレースのうち、 実際に荒れた (中穴+) 割合
  - **回収率**: virtualPnl の合計 ÷ 投入総額 (買い目数 × 5,000 円)
  - **期待値**: 1 レースあたりの平均 PnL
  - **最大連敗**: 連続して外したレース数の最大値
  - **見送り精度**: skip 判定したレースで実際に堅かった割合

#### UI
- 🔬 研究所タブ内に「バックテスト」 セクション追加
- 「現在の重み」 と「保存中の検証重み」 の指標を並列表示
- 「過去 7 日 / 14 日 / 30 日」 で期間切替

### 13.3 第 3 層: 本番 AI / 検証 AI 分離 (シャドーモード) (Round 187)

**ゴール**: 学習結果をいきなり本番反映しない。 まず検証 AI (= shadow) で試し、 一定期間 OK なら本番に昇格。

#### データ構造
- localStorage:
  - `mansyuWeights` (本番 — 全場共通) ← ユーザーに見せる予想に使用
  - `mansyuVenueWeights` (本番 — 場別)
  - **`mansyuShadowWeights` (検証 — 全場共通) ← 学習結果はまずここに入る**
  - **`mansyuShadowVenueWeights` (検証 — 場別)**
  - **`mansyuShadowHistory` (検証履歴 — 7-14 日後の評価結果)**

#### サイクル
1. **学習** (mansyuLearningAuto): 提案を生成
2. **検証 AI に保存**: `mansyuShadowWeights` に保存 (本番には反映しない)
3. **シャドー評価 (1 日 1 回 / 7-14 日継続)**: races を「本番重み」 と「検証重み」 の両方で評価し、 結果を比較
   - virtualPnl 比較: 検証 ≥ 本番 + 5% → 検証勝ち
   - 期待値比較: 検証 ≥ 本番 - 1% → 検証許容
4. **昇格判定**: 7-14 日後、 検証が本番より明確に良ければ本番に反映 / 同等以下なら破棄
5. ロールバック: 既存の Round 172.5 と同じ仕組みを本番側に適用

#### UI
- 🔬 研究所タブ「学習履歴」 にシャドー評価ステータス追加
- 「現在検証中の重み (試用 N 日目)」 「過去の昇格 / 破棄履歴」

## 14. 常時監視 + Web Push (アプリを閉じてても動く) (Round 189-194)

「ブラウザを開いていないと監視が止まる」 を解消する設計。 詳細は `docs/ROADMAP-ALWAYS-ON.md` に分離。 ここには SPEC として必須の合意事項だけを置く。

### 14.1 構成 (3 つの柱)

1. **Vercel Cron + Serverless API** — 5 分ごとに `/api/cron/scan-races` がサーバ側で 5 場を巡回。 15 分ごとに `/api/cron/finalize` が結果を取り込み収支確定。
2. **Supabase 主データ化** — `manfune_lab.skip_log` を真実の保管場所にし、 localStorage はキャッシュに格下げ。 サーバが書いた予想をクライアントが読む。
3. **Web Push** — Service Worker + VAPID。 激荒れスコア 85+ を初検出した瞬間にスマホへ Push。 通知タップ → 該当レースの大判カードを最上部に。

### 14.2 必須条件

- **Vercel Pro プラン課金 ($20/月)**: Hobby は 1 日 2 回しか cron が動かないので不可。
- **VAPID 秘密鍵を Vercel 環境変数のみで管理** — git にコミット禁止。
- **iOS Safari 利用時は PWA インストール導線** — Web Push 受信に必須 (16.4 以降)。

### 14.3 新規テーブル (manfune_lab スキーマ)

- `push_subscriptions` (user_id / endpoint / p256dh_key / auth_key / enabled) RLS 4 ポリシー
- `cron_log` (job / started_at / finished_at / races_scanned / alarms_found / notifications_sent / error)
- `skip_log` に `notified_at` カラム追加 (重複通知防止)

### 14.4 やらないこと (今回も対象外)

- 自動購入 / 自動ログイン (規約違反 + CLAUDE.md 禁止事項)
- ネイティブアプリ化 (App Store 申請)
- 競艇場映像のリアルタイム連携

### 14.5 完成イメージ

- スマホがロックされた状態で「🚨 戸田 9R 激荒れ警報 スコア 92 / 締切 8 分」 がプッシュ通知
- タップ → アプリが開く → 該当レースの大判カードが先頭表示 + 買い目 5,000 円配分も即見える
- レース終了後、 寝てる間に結果と仮想収支 (緑/赤) が反映済み

詳細フェーズ分け・コスト試算・リスクは `docs/ROADMAP-ALWAYS-ON.md` 参照。
